## Baseline After the 0.x Transition Release

### Public Tool Set

Python and TypeScript currently expose the same intended tool surface:

| Tool | Main data source | Status |
|------|------------------|--------|
| `search_prts` | PRTS MediaWiki API | Stable |
| `read_prts_page` | PRTS MediaWiki API | Stable |
| `get_operator_archives` | ArknightsGameData excel tables | Stable |
| `get_operator_voicelines` | ArknightsGameData excel tables | Stable |
| `get_operator_basic_info` | ArknightsGameData excel tables | Stable |

| `read_activity` | ArknightsStoryJson story files | Stable |

### Current Data Flow

- ArknightsGameData is synced from `3aKHP/ArknightsGameData` Release asset
  `zh_CN-excel.zip`, then extracted into the existing `gamedata` directory
  layout.
- ArknightsStoryJson is synced from `3aKHP/ArknightsStoryJson` Release asset
  `zh_CN.zip` and read directly as a zip archive.
- Docker and npm releases include CI-prewarmed fallback data.
- PyPI remains lightweight and depends on auto-sync or user-supplied paths.

## 1.0 Development Priorities

1. **Version alignment**
   - Move both implementations to the same 1.0 version line.
   - Document patch-version divergence rules.

2. **Dataset reader abstraction**
   - Introduce a shared conceptual API for directory-backed and zip-backed JSON
     reads.
   - Keep Python and TypeScript naming and behavior as close as their runtimes
     reasonably allow.

3. **Parser isolation**
   - Keep domain parsers focused on Arknights data semantics.
   - Remove direct filesystem/zip assumptions from operator and story parsing
     code.

4. **Compatibility and migration**
   - Preserve `GAMEDATA_PATH`, `STORYJSON_PATH`, `GITHUB_TOKEN`, and
     `GITHUB_MIRRORS` semantics unless a migration note is written first.
   - Keep current MCP tool parameters stable through the 1.0 line.

5. **Test and release confidence**
   - Expand shared fixture tests.
   - Verify published package contents for Docker and npm fallback data.
   - Keep PyPI packaging intentionally data-light.

## Deferred Feature Candidates

### Story summary by story key or stage

This remains optional for 1.0. The safer first version is source-backed only:

- `get_story_summary(story_key)` returns existing story summary text when
  available.
- `get_stage_story_summary(stage_id)` should wait until stage-to-story mapping
  is proven stable.

Avoid generated summaries in the initial 1.0 architecture line unless caching,
reproducibility, and dependency boundaries are explicitly designed.

### More operator data tools

Potential tools such as `get_operator_skills` and `get_operator_skins` become
easier after the reader abstraction lands. They should be considered after the
1.0 data layer is stable, not before.
