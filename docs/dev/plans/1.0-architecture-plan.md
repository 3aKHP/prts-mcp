# PRTS-MCP 1.0 Architecture Plan

_Draft date: 2026-05-03_

## Purpose

The 1.0 line should turn PRTS-MCP from two parallel implementations with
similar behavior into a stable, intentionally versioned product architecture.
The main work is data pipeline normalization, not new surface area.

Target release:

- Python `1.0.0`
- TypeScript `1.0.0`

Pre-release tags may be used for staged validation:

- `python/v1.0.0-alpha.1`
- `ts/v1.0.0-alpha.1`
- `python/v1.0.0-beta.1`
- `ts/v1.0.0-beta.1`

## Design Principles

1. **Keep user-facing behavior stable while internals move.**
   Current tools, environment variables, and fallback behavior should keep
   working unless a migration note is written first.

2. **Unify concepts before unifying physical storage.**
   The first step is a common reader abstraction. It is not necessary to decide
   immediately whether every dataset must be zip-backed or directory-backed at
   runtime.

3. **Make new tools data-driven.**
   Adding a future tool should usually mean declaring needed JSON paths and
   writing domain parsing logic, not adding another sync/storage path.

4. **Preserve package-size intent.**
   Docker and npm can include fallback data. PyPI should remain lightweight
   unless a separate packaging decision changes that policy.

## Current State

### Data Sources

| Dataset | Upstream mirror | Release asset | Runtime handling |
|---------|-----------------|---------------|------------------|
| Game data excel tables | `3aKHP/ArknightsGameData` | `zh_CN-excel.zip` | downloaded then extracted |
| Story data | `3aKHP/ArknightsStoryJson` | `zh_CN.zip` | read directly as zip |
| PRTS Wiki pages | PRTS MediaWiki API | none | network API |

### Current Mismatch

- Game data and story data now both arrive as Release archives.
- Only story data is read directly from the archive.
- Operator tools still read unpacked JSON files.
- Python and TypeScript have similar but separate sync, config, and parser
  code.

This is acceptable for the 0.x transition release, but it becomes expensive as
more data-backed tools are added.

## Target Architecture

The 1.0 data path should be split into four conceptual layers.

### 1. Dataset Source

Describes where a dataset comes from.

Common fields:

- dataset id, such as `gamedata.excel` or `story.zh_CN`
- GitHub owner/repo
- Release asset name
- expected internal paths
- cache metadata location
- fallback policy

Example conceptual specs:

```text
gamedata.excel
  owner: 3aKHP
  repo: ArknightsGameData
  asset: zh_CN-excel.zip
  required:
    zh_CN/gamedata/excel/character_table.json
    zh_CN/gamedata/excel/handbook_info_table.json
    zh_CN/gamedata/excel/charword_table.json
    zh_CN/gamedata/excel/story_review_table.json

story.zh_CN
  owner: 3aKHP
  repo: ArknightsStoryJson
  asset: zh_CN.zip
  required:
    zh_CN/gamedata/excel/story_review_table.json
    zh_CN/storyinfo.json
```

### 2. Local Storage

Describes what exists on disk.

Candidate storage types:

- `DirectoryStore`: reads JSON from a directory root.
- `ZipStore`: reads JSON entries from a zip file.
- `FallbackStore`: checks primary store first, then bundled store.

The code should avoid embedding assumptions such as:

- operator data must be unpacked
- story data must be zipped
- bundled data only exists in Docker

Those may remain true operationally, but parser code should not depend on them.

### 3. JSON Reader

Small runtime-neutral API shape:

```text
exists(path) -> bool
read_text(path) -> str
read_json(path) -> object
list(prefix) -> list[str]    optional for 1.0
describe() -> diagnostic string
```

Python and TypeScript implementations do not need identical class names, but
they should share behavior and test cases.

### 4. Domain Parser

Domain parsers should take a reader/store and know only Arknights data shapes.

Examples:

- `OperatorRepository`
  - reads `character_table.json`
  - reads `handbook_info_table.json`
  - reads `charword_table.json`

- `StoryRepository`
  - reads `story_review_table.json`
  - reads story chapter JSON by `story_key`
  - optionally reads `storyinfo.json`

MCP tool functions should call repositories, not raw files or zip entries.

## Proposed Module Layout

### Python

```text
python/src/prts_mcp/data/
  datasets.py        dataset specs and resolution
  stores.py          DirectoryStore, ZipStore, FallbackStore
  sync.py            Release sync against dataset specs
  operator.py        tool-facing formatter and repository
  story.py           tool-facing formatter and repository
```

### TypeScript

```text
ts/src/data/
  datasets.ts        dataset specs and resolution
  stores.ts          DirectoryStore, ZipStore, FallbackStore
  sync.ts            Release sync against dataset specs
  operator.ts        tool-facing formatter and repository
  story.ts           tool-facing formatter and repository
```

The layout can be adjusted during implementation, but both sides should keep
matching concepts.

## Compatibility Rules

### Must Preserve

- Existing MCP tool names.
- Existing required tool parameters.
- `GAMEDATA_PATH` disables gamedata auto-sync and points to an
  ArknightsGameData-compatible root.
- `STORYJSON_PATH` disables story auto-sync and points to a `zh_CN.zip`.
- `GITHUB_TOKEN` and `GITHUB_MIRRORS` behavior.
- Docker and npm fallback data support.

### May Change With Migration Notes

- Internal cache metadata file names.
- Internal module names.
- Whether gamedata is read extracted or directly from zip.
- Diagnostic wording when data is unavailable.

### Should Avoid in 1.0

- Changing tool names.
- Changing successful response format in a way that breaks existing consumers.
- Requiring network access when bundled or user-provided data exists.

## Implementation Phases

### Phase 0: Baseline Audit

- [ ] List every direct filesystem or zip read in Python and TypeScript.
- [ ] Identify all required JSON paths for existing tools.
- [ ] Document current fallback order for gamedata and story data.
- [ ] Confirm package content expectations for Docker, npm, and PyPI.

Exit criteria:

- A short audit note exists in `docs/dev/reports/`.
- No implementation changes yet.

### Phase 1: Reader Abstraction Skeleton

- [ ] Add directory-backed reader/store in Python.
- [ ] Add zip-backed reader/store in Python.
- [ ] Add fallback reader/store in Python.
- [ ] Add matching TypeScript stores.
- [ ] Add unit tests using the same minimal fixture content.

Exit criteria:

- Both implementations can read the same fixture from directory and zip forms.
- Existing tools still use old readers.

### Phase 2: Operator Parser Migration

- [ ] Move operator table loading behind the new reader.
- [ ] Preserve current cache behavior and cache invalidation.
- [ ] Keep existing output formatting.
- [ ] Add golden checks for `get_operator_basic_info`,
  `get_operator_archives`, and `get_operator_voicelines`.

Exit criteria:

- Existing operator tests pass.
- Same fixture produces equivalent Python/TS output.

### Phase 3: Story Parser Migration

- [ ] Move story metadata and chapter reads behind the new reader.
- [ ] Keep zip-backed story runtime behavior initially.
- [ ] Add tests for event listing, chapter listing, and story read from the
  reader abstraction.
- [ ] Decide whether `storyinfo.json` should become part of summary tooling.

Exit criteria:

- Existing story tests pass.
- Story tools do not directly instantiate zip readers in tool-facing code.

### Phase 4: Sync and Dataset Spec Consolidation

- [ ] Define dataset specs for gamedata and story data.
- [ ] Make sync code consume dataset specs rather than ad hoc Release specs.
- [ ] Normalize cache metadata behavior.
- [ ] Keep archive extraction where required for compatibility.
- [ ] Add package-content checks for CI release jobs.

Exit criteria:

- CI can verify Docker and npm bundled fallback data before publishing.
- PyPI packaging remains data-light by design.

### Phase 5: Versioning and Release Prep

- [ ] Bump Python and TypeScript to matching `1.0.0-alpha.1`.
- [ ] Add compatibility matrix to README.
- [ ] Add migration guide from the 0.x transition release.
- [ ] Verify tag workflows for pre-release versions.

Exit criteria:

- `python/v1.0.0-alpha.1` and `ts/v1.0.0-alpha.1` can be published from the
  same commit.

### Phase 6: Beta Hardening

- [ ] Freeze public tool parameters and primary output structures.
- [ ] Run full CI on clean environments.
- [ ] Review docs for Docker, npm, PyPI, and direct source usage.
- [ ] Decide whether optional summary tooling is included or deferred.

Exit criteria:

- No known migration blockers.
- 1.0 release notes are ready.

## Testing Plan

### Unit Tests

- Directory reader reads valid JSON.
- Zip reader reads valid JSON.
- Fallback reader prefers primary, falls back to bundled.
- Missing required file produces consistent diagnostics.
- Unsafe zip paths are rejected during extraction.

### Cross-Implementation Fixtures

Maintain minimal fixture data for:

- one operator with basic info, archives, and voice lines
- one story event with two chapters
- one missing-data case

The fixture should be intentionally small and hand-written so tests remain
fast and reviewable.

### Package/Release Tests

- Docker image contains fallback gamedata and `zh_CN.zip` when built by CI.
- npm package contains fallback data when published by CI.
- local npm pack without prewarm is allowed to contain only placeholders, but
  docs must say so.
- PyPI wheel does not contain large fallback data.

## Optional Story Summary Track

Story summary tooling should be treated as optional until the reader layer is
stable.

### Safe First Version

```text
get_story_summary(story_key)
```

Behavior:

- read existing source-provided summary text only
- return a clear "no summary available" message if absent
- do not generate summaries

Possible data sources:

- `zh_CN/storyinfo.json`
- per-story `storyInfo` fields

### Deferred Version

```text
get_stage_story_summary(stage_id)
```

This requires a stable mapping from stage id to story key. It should not block
1.0 unless the mapping can be proven from existing game tables.

### Out of Scope for Initial 1.0

- LLM-generated summaries
- remote summary services
- summary caches with invalidation rules

## Risks

### Risk: Over-abstracting Too Early

Mitigation:

- Keep the reader API tiny.
- Port existing tools before adding new tools.

### Risk: Breaking Custom Data Paths

Mitigation:

- Preserve current environment variable semantics.
- Add migration tests for `GAMEDATA_PATH` and `STORYJSON_PATH`.

### Risk: Python and TypeScript Drift

Mitigation:

- Keep matching fixture tests.
- Document any intentional implementation difference.
- Release alpha/beta tags from the same commit where possible.

### Risk: Package Size Surprises

Mitigation:

- Keep PyPI data-light.
- Check npm package contents in CI after prewarm.
- Check Docker image data presence in CI.

## Open Questions

- Should gamedata eventually be read directly from `zh_CN-excel.zip`, or should
  extraction remain the default storage mode behind the reader abstraction?
- Should cache metadata be per dataset under `archives/`, or centralized under
  a shared `data/cache/` path?
- Should pre-release tags publish to PyPI/npm automatically, or only build
  GitHub Release artifacts until beta?
- Is story summary tooling part of 1.0, or the first 1.x feature after the data
  layer stabilizes?
